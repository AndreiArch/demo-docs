The \{\{OcientHyperscaleDataWarehouse}} supports querying using the SQL syntax that follows ANSI SQL standards.

# Ocient SQL Syntax

@virtual

This code block shows the general syntax to perform SQL querying in \{\{Ocient}} and the order in which commands should go.&#x20;

For specific descriptions and syntax for the commands, see the respective SQL statement sections on this page.&#x20;

**Syntax**

```sql
[ WITH ... ]
SELECT ...
    [ EXCEPT(...) ]
[ FROM ...
    [ JOIN ... ] ]   
[ WHERE ... ]
[ GROUP BY ... 
    [ HAVING ... ] ]
[ ORDER BY ... ] 
[ LIMIT ... ]
[ OFFSET ... ] 
[ INTERSECT ... ]
[ EXCEPT ... ]
[ UNION ... ] ]         
[ USING ... ]
[ TRACE ... ] 
[ TAG ... ]
```

# Default Schema

Ocient identifies every table using a database and schema. For example, the fully qualified path to the `movies` table is `cinema.adventure.movies`, where `cinema` is the database and `adventure` is the schema.&#x20;

When you do not fully qualify a table name, the Ocient System uses a default schema. When you first log into the system, the default schema is your fully qualified username. You can change the default schema using the :Link[SET SCHEMA]{label="Commands Supported by the Ocient JDBC CLI Program " spaceId="hoqblJqChKOc8DvhBF0dZ" docId="zyOjXip3zkuVr9SkpIyxO" version="v2" docAnchorId="#anXvZ" loadingMethod="dynamic" overridedLabel="SET SCHEMA"} command.

## WITH

Assigns a name to a common table expression, allowing an auxiliary query to be used in the main query. This is helpful for breaking down complex queries into smaller parts.&#x20;

**Syntax**

```sql
WITH 
    <cte_name> [ ( <cte_column> [ ,... ] ) ] 
    AS ( <sub_query> )
<select_query>  
```

**Example**

In this example, the subquery calculates the average budget for all rows in the `movies` table. The main query uses that average to find all movies that spent more.&#x20;

```sql
WITH avg_budget_table (average_budget) AS (
    SELECT AVG(budget)
    FROM movies
)
SELECT title,
    budget,
    revenue
FROM movies,
    avg_budget_table
WHERE budget > avg_budget_table.average_budget;
```

### Using Lateral Column Aliases

Ocient SQL queries support lateral column aliases, meaning you can immediately reuse aliases for calculations in the same query as new inputs. Hence, you can simplify queries that normally require subqueries and common table expressions.&#x20;

These examples use the `products` table with these columns:

- `product_id` —  Product identifier as an integer
- `product_name` — Product name as a string
- `price` — Price as a floating point number

Create this table using the `CREATE TABLE` SQL statement.

```sql
CREATE TABLE products (
    product_id INT,
    product_name VARCHAR(100),
    price DECIMAL(10, 2)
    ); 
```

Insert four records into the `products` table.

```sql
INSERT INTO products (product_id, product_name, price) VALUES
    (1, 'Laptop', 1000.00),
    (2, 'Tablet', 500.00),
    (3, 'Smartphone', 800.00),
    (4, 'Monitor', 300.00);
```

Create a query to determine prices after discounts and taxes by using a common table expression subquery `DiscountedPrices`. Use the `WITH` keyword to create the subquery.

```sql
WITH DiscountedPrices AS (
    SELECT 
        product_id,
        product_name,
        price,
        price * 0.90 AS discounted_price,
        price * 0.90 * 1.05 AS total_price_after_tax
    FROM 
        products
)
SELECT 
    product_id,
    product_name,
    price,
    discounted_price,
    total_price_after_tax
FROM 
    DiscountedPrices;
```

Lateral aliases allow the same calculations to be packaged in a single query.&#x20;

This simpler query is essentially the same as the longer common table expression example, but the logic is condensed because you can reference the `discounted_price` alias immediately to calculate the `total_price_after_tax` value in the same query.&#x20;

```sql
SELECT 
    product_id,
    product_name,
    price,
    price * 0.90 AS discounted_price,
    discounted_price * 1.05 AS total_price_after_tax
FROM 
    products;
```

      Join conditions can use any Boolean expression, including `=`, `!=`, `<`, `>`, `=>`, and `<=`
    </td>
  </tr>
  <tr>
    <td align="left">
      `table2_column`
    </td>
    <td align="left">
      A column from the `table_reference` not used for `table1_column` &#x20;
      The `JOIN` operation uses this column to match rows based on the `table1_column` to combine data for the result set.&#x20;
    </td>
  </tr>
</table>

### Types of Join Operations ( `<join_operation>` )

Ocient supports the following types of `JOIN` operations.&#x20;

**Syntax**

```sql
<join_operation> ::=   
    { INNER JOIN
    | LEFT [ OUTER ] JOIN
    | RIGHT [ OUTER ] JOIN
    | FULL [ OUTER ] JOIN
    | CROSS JOIN
    | SEMI JOIN 
    | ANTI JOIN } 
```

### **JOIN Type Descriptions**

<table columnWidths="181,100" selectedRows="" selectedTable="false" isTableHeaderOn="true" selectedColumns="">
  <tr>
    <td align="left">
      **JOIN Type**
    </td>
    <td align="left">
      **Description**
    </td>
  </tr>
  <tr>
    <td align="left">
      `INNER JOIN`
    </td>
    <td align="left">
      Returns all rows that exist in both tables. This join type is the default type of join.
    </td>
  </tr>
  <tr>
    <td align="left">
      `LEFT OUTER JOIN`
    </td>
    <td align="left">
      Returns all rows from the first table regardless of whether there are matching rows in the second table.&#x20;
      `LEFT JOIN` performs the same operation as `LEFT OUTER JOIN`. &#x20;
    </td>
  </tr>
  <tr>
    <td align="left">
      `RIGHT OUTER JOIN`
    </td>
    <td align="left">
      Returns all rows from the second table regardless of whether there are matching rows in the first table.
      &#x20;`RIGHT JOIN` performs the same operation as `RIGHT OUTER JOIN`. &#x20;
    </td>
  </tr>
  <tr>
    <td align="left">
      `FULL OUTER JOIN`
    </td>
    <td align="left">
      Returns all matched and unmatched rows.
      `FULL JOIN` performs the same operation as `FULL OUTER JOIN`. &#x20;
    </td>
  </tr>
  <tr>
    <td align="left">
      `CROSS JOIN`
    </td>
    <td align="left">
      Returns the Cartesian product of the joined tables. Each row in the first table is joined with all rows in the second table. The result set contains the number of rows in the first table multiplied by the number of rows in the second table.
    </td>
  </tr>
  <tr>
    <td align="left">
      `SEMI JOIN`
    </td>
    <td align="left">
      Returns the rows from the first table that have at least one match in the second table. No columns from the second table are available in the query.
    </td>
  </tr>
  <tr>
    <td align="left">
      `ANTI JOIN`
    </td>
    <td align="left">
      Returns the rows from the first table that have no matches in the second table. No columns from the second table are available in the query.
    </td>
  </tr>
</table>

<hint type="info">
  By default, `JOIN` statements that involve subqueries can operate laterally if necessary. This behavior allows subqueries to reference joined columns from the preceding items included in the `FROM` clause.   

  For example, both parts of this join operation use subqueries that reference table `x`. The `LATERAL` keyword is optional; the join operation acts the same regardless of whether it is included.&#x20;

  <BlockQuote>
    SELECT \* FROM
    &#x9;(
    &#x9;SELECT
    &#x9;	1 AS cx1,
    &#x9;	2 AS cx2) AS x
    INNER JOIN LATERAL (
    &#x9;SELECT
    &#x9;	x.cx1 AS cy1,
    &#x9;	x.cx2 AS cy2) AS y&#x20;
    ON
    &#x9;x.cx1 = y.cy1;
  </BlockQuote>

  Lateral joins are primarily useful when a cross-referenced column is necessary for computing the rows to join. A common application is providing an argument value for a set-returning function. 
</hint>

### EXISTS

An `EXISTS` clause used in a `WHERE` statement evaluates if a subquery returns any rows.&#x20;

For each row that the database computes in the outer query, if the subquery returns at least one row, the `EXISTS` clause evaluates to `true`. In this outcome, the outer query returns its row.&#x20;

If the subquery returns zero rows, the `EXISTS` clause evaluates to `false`, and the outer query excludes those rows.&#x20;

The `NOT EXISTS` clause performs the opposite Boolean logic. In this case, the outer query returns rows only when the subquery has no matches.&#x20;

**Examples**

These examples use two tables, one for company departments and the other for employees who are assigned to those departments.&#x20;

Create the `departments` table for department data.

```sql
CREATE TABLE departments (
    id INT,
    name VARCHAR(50)
);
```

Create the `employees` table for employee data.

```sql
CREATE TABLE employees (
    id INT,
    name VARCHAR(50),
    department_id INT
);
```

Insert department data for three departments, `HR`, `IT`, and `Marketing`, into the `departments` table.

```sql
INSERT INTO departments (id, name) VALUES
    (1, 'HR'),
    (2, 'IT'),
    (3, 'Marketing');
```

Insert employee data for four employees, `Alice`, `Bob`, `Charlie`, and `David`, into the `employees` table.&#x20;

```sql
INSERT INTO employees (id, name, department_id) VALUES
    (1, 'Alice', 1),
    (2, 'Bob', 2),
    (3, 'Charlie', 2),
    (4, 'David', 4);
```

**Find Employees Belonging to an Existing Department**

This example uses an `EXISTS` clause to find any names from the `employees` table who are assigned to a department listed in the `departments` table.&#x20;

```sql
SELECT name
FROM employees AS e
WHERE EXISTS (
    SELECT *
    FROM departments AS d
    WHERE e.department_id = d.id
);
```

The query returns all employees except `David`, who is assigned to a department that does not exist in the `departments` table.&#x20;

*Output*

```none
Bob 
Charlie
Alice
```

**Find Departments That Have Employees**

This query returns departments that have at least one employee. The subquery uses `SELECT 1` to check whether any matching rows exist in the `employees` table. The output is the same if the query uses `SELECT *` instead.&#x20;

```sql
SELECT name 
FROM departments AS d
WHERE EXISTS (
    SELECT 1 
    FROM employees AS e 
    WHERE e.department_id = d.id
);
```

The query results exclude the `Marketing` department because no employees belong to it.&#x20;

*Output*

```none
HR        
IT 
```

**Filter Departments Based on an Uncorrelated Table**

In this example, the outer query filters the `departments` table where `id != 3` (excluding the `Marketing` department). The subquery checks if there is at least one employee with a department identifier less than `4`.

The query is uncorrelated because the subquery does not reference the `departments` table.

```sql
SELECT * 
FROM departments d
WHERE d.id != 3 
AND EXISTS (
    SELECT * 
    FROM employees e 
    WHERE e.department_id < 4
);
```

*Output*

```none
HR        
IT 
```

**Find Employees Without a Valid Department**

This example uses a `NOT EXISTS` clause. The example returns only employees who are assigned to a department identifier `department_id` not listed in the `departments` table.&#x20;

```sql
SELECT name
FROM employees AS e
WHERE NOT EXISTS (
    SELECT *
    FROM departments AS d
    WHERE e.department_id = d.id
);
```

*Output:&#x20;*`David`

### Array Filters

A filter expression can use the functions `FOR_SOME()` and `FOR_ALL()` to apply a predicate against all values of an input array. 

Array filter functions have the following rules:&#x20;

- They can only evaluate array types. 
- They can go on either the left or right side of a boolean comparison expression, but not both sides at the same time. 
- They must directly use a boolean comparison operator.
- They cannot use the `SOME` and `ALL` SQL keywords. 

**Filter Behavior with Empty Arrays**

Array filter functions have unique behavior when evaluating empty arrays.&#x20;

- `FOR_ALL()` evaluates an empty array as `TRUE`.&#x20;
- `FOR_SOME()` evaluates an empty array as `FALSE`.

If empty arrays must be evaluated for a different result, you can use the `ARRAY_LENGTH` function to specify a minimum array length. For example, this statement would evaluate an array as `TRUE` only if it is not empty and all values matched `%ocient%`:

```sql
ARRAY_LENGTH(array_col) > 0 AND FOR_ALL(array_col) LIKE '%ocient%
```

For details about array functions, see the :Link[Array Functions and Operators]{docId="XUK0Z8dmXPgMOgSzPDW6w" label="Array Functions and Operators" spaceId="C0TRtFmQ9oOqRtkCB9A9i" version="v2" docAnchorId loadingMethod="dynamic" overridedLabel} page.&#x20;

**Filter Behavior with NULL rows**

Filtering with array functions can have different outcomes depending on whether the array row is NULL or if the values contained in the array are NULL.&#x20;

If `FOR_SOME()` or `FOR_ALL()` evaluate a NULL row (the row itself is NULL, not that the array contains NULL values), then the result is always `FALSE`. 

To check an array for the presence of NULL values, you must use the `IS NULL` operator. For example:&#x20;

```sql
FOR_SOME(array_col) IS NULL
```

Array comparison operators, such as `@>`, `<@`, and `&&` do not adhere to Boolean logic for NULL values. For details, see the :Link[Array Functions and Operators]{docId="XUK0Z8dmXPgMOgSzPDW6w" label="Array Functions and Operators" spaceId="C0TRtFmQ9oOqRtkCB9A9i" version="v2" docAnchorId loadingMethod="dynamic" overridedLabel} page.

## EXCEPT

The `EXCEPT` keyword returns the result set of a first `SELECT` query minus any matching rows from a second `SELECT` query.

The `EXCEPT` keyword can also be used to exclude specific columns from a `SELECT *` query. For information on that alternate usage, see the :Link[SELECT]{docId="Nw9vaVKey2v75MoXm_muO" label="General SQL Syntax" spaceId="IVveQLbBJeU-r82NRG4-0" version="v2" docAnchorId="#XS3Ek" loadingMethod="dynamic" overridedLabel="SELECT"} syntax and example.   &#x20;

## USING

Overrides various system configurations for processing a specified query.&#x20;

The `USING` keyword is required for only the first query override, not for subsequent ones.&#x20;

For descriptions of the supported query configurations, see the parameter table below.&#x20;

**Syntax**

```sql
<select_query>
USING 
    [ SCHEDULING_PRIORITY = priority_value ] 
    [ MAX_ROWS_RETURNED = max_rows ]
    [ MAX_ELAPSED_TIME = max_elapsed_time ] 
    [ MAX_TEMP_DISK_USAGE = max_temp_disk_usage ] 
    [ CACHE_MAX_TIME = cache_max_time ]
    [ CACHE_MAX_BYTES = cache_max_bytes ]
    [ SERVICE CLASS service_class ] 
```

**Parameters**

<table columnWidths="331,331" selectedRows="" selectedTable="false" isTableHeaderOn="true" selectedColumns="">
  <tr>
    <td align="left">
      **Parameter**
    </td>
    <td align="left">
      **Description**
    </td>
  </tr>
  <tr>
    <td align="left">
      `<select_query>`
    </td>
    <td align="left">
      A valid `SELECT` query.
    </td>

      `priority_value`
    </td>
    <td align="left">

    </td>

## TRACE

An optional clause that executes the query, but discards the original result set. Instead, `TRACE` returns a result set of tracing data that describe the execution of the query.

Append the `TRACE` SQL statement to the end of a query. The statement can include optional parameters to control its frequency and level of detail.&#x20;

To understand the results of this statement, see :Link[TRACE Results]{docId="Nw9vaVKey2v75MoXm_muO" label="General SQL Syntax" spaceId="IVveQLbBJeU-r82NRG4-0" version="v2" docAnchorId="#v0BJ8" loadingMethod="dynamic" overridedLabel="TRACE Results"}.&#x20;

<hint type="info">
  Trace queries have the same effect to the workload as a query run without the clause.
</hint>

<hint type="warning">
  Contact Ocient Support for guidance in using the TRACE SQL statement.
</hint>

**Syntax**

```sql
<select_query> TRACE 
    [ FREQUENCY frequency_int ] 
    [ RESOLUTION resolution_int ] 
```

## TAG

An optional clause that adds one or more tags to the SQL query. You can also add tags to a common table expression in the `WITH` clause. After you define tags, you can find them in the `sys.queries` and `sys.completed_queries` system catalog tables.

**Syntax**

```sql
<select_query> <tag> [ ... ]

<tag> ::= TAG <tag_identifier> 
```

**Parameters**

<table isTableHeaderOn="true" selectedColumns="" selectedRows="" selectedTable="false" columnWidths="331,331">
  <tr>
    <td align="left">
      **Parameter**
    </td>
    <td align="left">
      **Description**
    </td>
  </tr>
  <tr>
    <td align="left">
      `<select_query>`
    </td>
    <td align="left">
      A valid `SELECT` query.&#x20;
    </td>
  </tr>
  <tr>
    <td align="left">
      `tag_identifier`
    </td>
    <td align="left">
      Identifier of the tag for the query. Enclose the identifier in double quotes if it contains any special characters, such as spaces.&#x20;
    </td>
  </tr>
</table>

**Examples**

**SQL Query with One Tag**

Select the count of a table with one row and tag the query with the name `count1`.

```sql
SELECT COUNT(*) FROM sys.dummy1 TAG count1;
```

**SQL Query with Multiple Tags**

Select the device model and tag this query with two names `device_model` and `phones`.

```sql
SELECT device_model 
FROM system.adtech_flat 
LIMIT 1 
TAG device_model 
TAG phones;
```

**Common Table Expression with One Tag**

Define a common table expression with the `average_budget` tag. You can also add another tag for the whole query `overall_budget`.

```sql
WITH avg_budget_table (average_budget) AS (
    SELECT AVG(budget)
    FROM movies
    TAG average_budget
)
SELECT title,
    budget,
    revenue
FROM movies,
    avg_budget_table
WHERE budget > avg_budget_table.average_budget
TAG overall_budget;
```

## String Literals and Escape Sequences

All string literals in SQL statements must be enclosed in single quotes. To use a single quote within a string, you can use another single quote as an escape, `''`*.&#x20;*

For other escape sequences, include an `e` character before the string literal, i.e., before the opening single quote. This directs the system to recognize escape sequences in the string literal. This means:&#x20;

- All single `\` characters in the string now escape themselves.
- Any subsequent character after the `\` is also escaped if it matches an escape sequence (see the table). &#x20;

If you use escape sequences, be prepared that you might need to alter strings that include `\`, such as directory paths.&#x20;

**Examples**

These examples show how the system interprets strings with and without escape sequences.&#x20;

**String Without an Escape Sequence**

This example selects the simple string literal `'\my\directory\path'` that does not use escape sequences.&#x20;

```sql
SELECT '\my\directory\path';
```

*Output:&#x20;*`\my\directory\path`

**String With an Escape Sequence**

This example selects the simple string literal `'\my\directory\path'` and uses an escape sequence. The result omits the backslashes.&#x20;

```sql
SELECT e'\my\directory\path';
```

*Output:&#x20;*`mydirectorypath`

**String With an Escape Sequence to Retain the Backslashes**

This example uses the `'\\my\\directory\\path'` string with an escape sequence to include backslashes in the result. &#x20;

```sql
SELECT e'\\my\\directory\\path';
```

*Output:&#x20;*`\my\directory\path`

**Escape Sequences with Regular Expressions**

<hint type="info">
  Be careful using escape sequences with strings that also use regular expressions. Escape sequences in SQL syntax override the escape sequences of regular expressions.&#x20;
</hint>

In this example, the :Link[REGEXP\_SUBSTR]{label="Character / Binary Functions" spaceId="_RIVPG6C91qqB76KHDBSV" docId="ja8cOnT33toNX-KTRuedJ" version="v2" docAnchorId="#z_x9P" loadingMethod="dynamic" overridedLabel="REGEXP_SUBSTR"} function uses the regular expression `\w+` to match any word characters.&#x20;

```sql
SELECT REGEXP_SUBSTR(
    'abcdefghijklmnopqrstuvwxyz', 
    '\w+'
    );
```

*Output:&#x20;*`abcdefghijklmnopqrstuvwxyz`

The function behaves differently if the regular expression is an escape sequence because it overrides the `\` character. The regular expression searches only for the character `w`.&#x20;

```sql
SELECT REGEXP_SUBSTR(
    'abcdefghijklmnopqrstuvwxyz', 
    e'\w+'
    );
```

*Output:&#x20;*`w`
